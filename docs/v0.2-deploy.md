# Self-Host Deployment Guide (v0.2)

How to deploy `@netlinksinc/odoo-mcp` in HTTP mode so Claude Desktop's "Add custom connector" UI and Claude Cowork can reach it. Single-tenant — one deploy serves one Odoo, credentials configured at deploy time.

---

## 1. Fly.io (canonical)

Fly.io is the primary supported target. ~5 minutes from `git clone` to a public HTTPS endpoint.

### Prereqs

- A Fly.io account (free tier covers a v0.2 demo)
- `fly` CLI installed (`brew install flyctl` on macOS)
- This repo cloned locally, or a fork

### Steps

```bash
# 1. Auth and init
fly auth login
fly launch --no-deploy

# Edit fly.toml if needed — change `app = "odoo-mcp"` to a unique name.

# 2. Generate a bearer token (this is what MCP clients will use)
openssl rand -hex 32
# → e.g. 7c3a91b4ee...d8 (64 hex chars). Copy it; you'll need it twice.

# 3. Set Odoo + bearer secrets
fly secrets set \
  ODOO_URL=https://your.odoo.example.com \
  ODOO_DB=your_database \
  ODOO_USERNAME=mcp_user \
  ODOO_API_KEY=your_odoo_api_key \
  MCP_BEARER_TOKEN=<the-hex-token-from-step-2>

# 4. Deploy
fly deploy

# 5. Verify
curl https://<app>.fly.dev/health
# Expected: {"ok":true,"mode":"http","odoo_url":"https://...","odoo_db":"...","started_at":"...","probe_ok":true}
```

If `/health` returns `{"ok":false,"probe_ok":false}` the server is reachable but the Odoo probe failed — check `fly logs` for the specific Odoo error and confirm the `ODOO_*` secrets are correct.

## 2. Claude Desktop — Add Custom Connector

In Claude Desktop's **Settings → Connectors → Add custom connector**:

| Field | Value |
|---|---|
| **Name** | `odoo` (or whatever you want shown in the connector list) |
| **Remote MCP server URL** | `https://<app>.fly.dev/mcp` |
| **Advanced → Custom headers** | `Authorization: Bearer <your-token-from-step-2>` |

Restart Claude Desktop (full quit, ⌘Q on macOS). The connector should appear in the MCP slider of a new conversation with 10 tools and 7 resources.

---

## 3. Claude Cowork

In Cowork's MCP server settings (the exact UI varies by Cowork version):

- **Remote MCP server URL:** `https://<app>.fly.dev/mcp`
- **Bearer token:** `<your-token-from-step-2>`

Once added, Cowork can call any of the 10 tools from any conversation.

## 4. Alternative Hosts

The same Dockerfile and entry point work anywhere Node 22 runs with TLS termination at the edge.

### Render

1. Create a new Web Service from this repo
2. Build Command: (auto-detected from Dockerfile) — leave blank
3. Start Command: (auto-detected from Dockerfile's ENTRYPOINT) — leave blank
4. Set the 5 env vars in the Render dashboard: `ODOO_URL`, `ODOO_DB`, `ODOO_USERNAME`, `ODOO_API_KEY`, `MCP_BEARER_TOKEN`
5. Deploy. Render terminates TLS automatically.

### Railway

```bash
railway login
railway up
# Then set the 5 env vars via railway dashboard or `railway variables set`
```

### Hostinger VPS / generic Linux

For full control on a Linux box:

```bash
# On the VPS, as a non-root user:
curl -fsSL https://get.pnpm.io/install.sh | sh
nvm install 22
git clone https://github.com/<you>/odoo-mcp.git
cd odoo-mcp
pnpm install
pnpm -r build

# Create the env file
cat > /etc/odoo-mcp.env <<EOF
MODE=http
MCP_PORT=3000
MCP_BEARER_TOKEN=<your-hex-token>
ODOO_URL=https://your.odoo.example.com
ODOO_DB=your_database
ODOO_USERNAME=mcp_user
ODOO_API_KEY=your_odoo_api_key
EOF
sudo chmod 600 /etc/odoo-mcp.env

# Systemd unit at /etc/systemd/system/odoo-mcp.service
sudo tee /etc/systemd/system/odoo-mcp.service > /dev/null <<EOF
[Unit]
Description=Odoo MCP server
After=network-online.target

[Service]
Type=simple
EnvironmentFile=/etc/odoo-mcp.env
ExecStart=/home/<user>/.nvm/versions/node/v22.0.0/bin/node /home/<user>/odoo-mcp/packages/odoo-mcp/dist/bin.js
Restart=on-failure
User=<user>

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl enable --now odoo-mcp
sudo systemctl status odoo-mcp

# TLS termination via nginx + Let's Encrypt
sudo apt install nginx certbot python3-certbot-nginx
# /etc/nginx/sites-available/odoo-mcp (proxy :443 → :3000 with appropriate headers)
sudo certbot --nginx -d odoo-mcp.yourdomain.com
```

Important: TLS termination must happen at the proxy layer (nginx, Caddy, Traefik). The Node process does NOT handle TLS itself. If clients connect over `http://` instead of `https://`, the bearer token and all Odoo data flow in plaintext.

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| `/health` returns `{"ok":false}` | Odoo auth succeeded but capability probe partially failed | `fly logs` to see which probe field errored. Common: user lacks read access on `ir.module.module` etc. |
| `/mcp` returns 401 on every request | Bearer token mismatch between server and client | Compare the value in Claude Desktop's custom-header field with `fly secrets list` (the value isn't shown — re-set if uncertain) |
| Server boots then immediately exits | One of the 5 env vars is missing or the API key is wrong | `fly logs` shows the structured `config_error` or `startup_error` JSON line |
| Claude Desktop shows the connector but no tools | Cached schema — restart Claude Desktop fully (⌘Q, not just close window) | |
| Need Odoo Python tracebacks | Diagnostic info gated by env var | `fly secrets set ODOO_MCP_DEBUG=1 && fly deploy` |

## What's NOT in v0.2

These belong to a future v0.3+ or are explicitly out of scope:

- OAuth 2.1 / multi-tenant credential storage (single-tenant only)
- Hosted-by-NETLINKS instance (you self-host)
- Anthropic connector directory listing
- Built-in TLS (delegated to the host)
- Horizontal scaling (single Node process by design)

See `README.md` Roadmap section for the broader plan.
