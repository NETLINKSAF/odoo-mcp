# v0.2.1 — OAuth 2.1 + per-user identity

This release replaces the single static bearer token of v0.2.0 with a full
OAuth 2.1 Authorization Code flow (PKCE S256). Each end-user authenticates
with their own Odoo credentials, receives their own access token, and tool
calls execute under their per-user `OdooClient` and identity.

> **stdio mode is unchanged.** If you run the server with `MODE=stdio`
> (default), nothing in this guide applies. `ODOO_USERNAME` and `ODOO_API_KEY`
> work exactly as before; none of the OAuth env vars are required.

---

## Overview

| | v0.2.0 | v0.2.1 |
|---|---|---|
| Authentication | Static bearer token (`MCP_BEARER_TOKEN`) | OAuth 2.1 Authorization Code + PKCE |
| Identity | Single shared identity | Per-user identity (email) |
| Tool calls | All from the startup `ODOO_USERNAME` | From each user's own Odoo creds |
| Credentials at rest | None | AES-256-GCM encrypted in JSON store |
| Tokens at rest | The bearer itself, in env | SHA-256 hashes only — raw tokens never persisted |
| Token revocation | Restart with new env value | `odoo-mcp users revoke <email>` (online) |

The OAuth dance is automatic in clients that implement RFC 8414 metadata
discovery + Dynamic Client Registration (RFC 7591). For clients that don't,
the bundled `odoo-mcp auth` subcommand walks the user through the dance and
prints the resulting access token.

## New environment variables (HTTP mode only)

All variables below are required when `MODE=http`. Generate the secrets ONCE
per deployment and keep them stable across restarts.

```bash
# Encryption key for user credentials at rest (32 raw bytes, base64-encoded)
MCP_ENCRYPTION_KEY=$(openssl rand -base64 32)

# Admin API password (used as Bearer for /admin/users)
MCP_ADMIN_PASSWORD=$(openssl rand -hex 32)

# Where the encrypted user store lives. The file is created on first write
# with mode 0o600. The parent directory must be writable by the server.
MCP_USER_STORE_PATH=/var/lib/odoo-mcp/users.json

# Optional. Public URL the OAuth metadata document advertises. Required
# whenever the server runs behind a TLS proxy (mcp.example.com → :3000).
# Falls back to the request's Host header, then localhost:<port>.
MCP_PUBLIC_URL=https://mcp.example.com

# Unchanged from v0.2.0
MODE=http
MCP_PORT=3000
```

If you generate `MCP_ENCRYPTION_KEY` with `openssl rand -base64 32` the
server will accept it. Any other length or format will exit with a config
error at startup.

In `stdio` mode the four new vars above are NOT required. `loadConfig`
gates the validation per mode.

## Removed variables

- **`MCP_BEARER_TOKEN`** — silently ignored. The /mcp endpoint now requires
  an OAuth-issued access token; static bearer tokens are no longer honored.
  Remove this variable from your deployment when you upgrade.

## Admin setup

Before any end-user can authenticate, an admin must add their email to the
allowlist. The CLI ships with the server:

```bash
# Allow a user
odoo-mcp users allow user@example.com \
  --url https://mcp.example.com

# List allowed users
odoo-mcp users list \
  --url https://mcp.example.com

# Revoke a user (invalidates their tokens, evicts cache, removes from store)
odoo-mcp users revoke user@example.com \
  --url https://mcp.example.com
```

Each command authenticates to the server with `MCP_ADMIN_PASSWORD` as the
Bearer token, so it can be run from anywhere with HTTPS access — not just
the server host.

Adding a user only puts their email on the allowlist. The user themselves
still has to complete the OAuth dance (below) before they can call tools.

## End-user flow — Claude Desktop / Cursor / any DCR-capable client

Clients that implement the MCP OAuth profile (RFC 8414 discovery + RFC 7591
DCR) handle the entire dance automatically. The user only ever sees the
consent screen in their browser.

```
User adds the server URL to Claude Desktop
  → Claude Desktop fetches  /.well-known/oauth-authorization-server
  → Claude Desktop POSTs    /oauth/register   (DCR)
  → Claude Desktop opens    /oauth/authorize  in the browser
  → User enters email + Odoo API key on the consent page
  → Server redirects back to Claude Desktop with ?code=…&state=…
  → Claude Desktop POSTs    /oauth/token      (PKCE verifier)
  → Claude Desktop stores the access token and uses it for /mcp requests
```

No manual configuration. No copying tokens.

---

## End-user flow — Claude Code (manual bearer)

Claude Code doesn't run the OAuth dance itself. Use the bundled `auth`
subcommand to run it from the terminal:

```bash
# Step 1 — run the dance
odoo-mcp auth https://mcp.example.com

# Step 2 — a browser tab opens at the consent page.
#          Enter your email + Odoo API key. Submit.

# Step 3 — the terminal prints the issued token:
Access token: a1b2c3d4e5f6…  (64 hex chars)

# Step 4 — register the server in Claude Code with that bearer:
claude mcp add \
  --header "Authorization: Bearer a1b2c3d4e5f6…" \
  https://mcp.example.com/mcp
```

The token is valid until you (or an admin) revoke it. There is no
explicit expiration — the v0.2.1 token store has no TTL, only a per-user
FIFO cap of 10 tokens, which lets a user maintain a few parallel
clients without losing access.

## Key rotation

`MCP_ENCRYPTION_KEY` rotation is destructive to the user store. There is
no key-versioned re-wrap path. To rotate:

1. Stop the server.
2. Replace `MCP_ENCRYPTION_KEY` in your deployment.
3. Start the server.
4. On startup, `load()` will iterate the user records and attempt to
   decrypt each `encrypted_api_key`. Every record encrypted under the old
   key will fail decryption and be logged with the record's email; the
   record is then SKIPPED (the rest of the store loads normally).
5. Every affected user must re-run `odoo-mcp auth` to re-register their
   Odoo API key.

For routine secret hygiene, rotate `MCP_ADMIN_PASSWORD` freely — it has
no on-disk dependencies. Just push a new env and restart.

## Revoking users

```bash
odoo-mcp users revoke user@example.com \
  --url https://mcp.example.com
```

This runs SYNCHRONOUSLY before the HTTP response:

1. All tokens belonging to the user are deleted from the store.
2. The user's cached `OdooClient` (if any) is evicted from the client cache.
3. The user record is removed from the store.
4. The store is flushed to disk (atomic rename, mode 0o600).

Any in-flight request that holds a stale token completes; the next request
with that token returns 401.

## Security notes

- **Tokens at rest**: only the SHA-256 hash of each issued token is written
  to `MCP_USER_STORE_PATH`. The raw token never touches disk.
- **API keys at rest**: each user's Odoo API key is encrypted under
  `MCP_ENCRYPTION_KEY` with AES-256-GCM (fresh IV per encryption, 16-byte
  auth tag). Decryption failures (wrong key, tampered ciphertext) are
  logged and the affected record is skipped — never crashes the server.
- **File permissions**: the user store is written with `chmod 0o600` on
  every atomic rename.
- **PKCE**: only S256 is accepted. Authorization codes are 32 hex chars,
  single-use, 10-minute TTL.
- **DCR**: rate-limited to 10 registrations per source IP per 60s; capped
  at 1000 total registered clients.
- **Admin API**: bearer-compared in constant time against
  `MCP_ADMIN_PASSWORD`. 6th failed attempt from the same IP within 60s
  returns 429.
- **Request body cap**: 64 KiB on all routes. Larger bodies return 413.

## Upgrading from v0.2.0

1. Set the 4 new env vars (`MCP_ENCRYPTION_KEY`, `MCP_ADMIN_PASSWORD`,
   `MCP_USER_STORE_PATH`, optionally `MCP_PUBLIC_URL`).
2. Remove `MCP_BEARER_TOKEN` (it's now ignored).
3. Restart the server.
4. Run `odoo-mcp users allow <email>` for each end-user.
5. Each end-user runs `odoo-mcp auth <url>` and reconfigures their client
   with the printed bearer.

That's it. There's no data migration — the old static-token setup leaves
no on-disk state to clean up.

## stdio mode (unchanged)

```bash
# Same as v0.2.0
MODE=stdio  # or unset
ODOO_URL=https://your-odoo
ODOO_DB=your_db
ODOO_USERNAME=admin
ODOO_API_KEY=…
```

`MCP_ENCRYPTION_KEY` / `MCP_ADMIN_PASSWORD` / `MCP_USER_STORE_PATH` are
NOT validated when `MODE=stdio`. The server boots with a single Odoo
identity (the startup `ODOO_USERNAME`), exactly as in v0.2.0.
