## Architecture Overview

Two npm packages in a pnpm workspace. `odoo-client` is a standalone JSON-RPC client for Odoo 19 with zero MCP dependency. `odoo-mcp` imports `odoo-client` and wraps it in an MCP server (stdio transport). Claude Desktop/Code connects to `odoo-mcp` via stdio; `odoo-mcp` issues JSON-RPC calls to the Odoo 19 instance.

- **Purpose**: Stateless JSON-RPC client for Odoo 19. Handles auth, all ORM calls, execute, report, action, fields_get, and probe.
- **Covers**: US-2, US-3, US-4, US-5, US-6, US-7, US-8

- **Interface**:

```typescript
export class OdooClient {
  constructor(config: OdooConfig);

  /** Authenticate and return session. Throws OdooAuthError on failure. */
  authenticate(): Promise<OdooSession>;

  /** ORM methods — all accept optional context override */
  searchRead(model: string, domain: Domain, fields: string[], options?: {
    limit?: number; offset?: number; order?: string; context?: Context;
  }): Promise<OdooRecord[]>;

  read(model: string, ids: number[], fields: string[], context?: Context): Promise<OdooRecord[]>;

  create(model: string, values: Record<string, unknown> | Record<string, unknown>[],
    context?: Context): Promise<number | number[]>;

  write(model: string, ids: number[], values: Record<string, unknown>,
    context?: Context): Promise<boolean>;

  unlink(model: string, ids: number[], context?: Context): Promise<boolean>;

  searchCount(model: string, domain: Domain, context?: Context): Promise<number>;

  /** Generic execute_kw */
  execute(model: string, method: string, args: unknown[], kwargs?: Record<string, unknown>,
    context?: Context): Promise<unknown>;

  /** Report engine — abstracts protocol differences */
  runReport(reportId: number | string, docIds: number[],
    context?: Context): Promise<{ content: string; contentType: string }>;

  /** Action invocation */
  callAction(model: string, ids: number[], actionName: string,
    context?: Context): Promise<unknown>;

  /** Introspection */
  fieldsGet(model: string, attributes?: string[],
    context?: Context): Promise<Record<string, Record<string, unknown>>>;

  /** Capability probe — runs all sub-queries, never throws for individual failures */
  probe(): Promise<ProbeResult>;
}
```

## Error Handling Strategy

### Error Taxonomy

| Error Class | Odoo Fault `data.name` | `error_type` Value | Recovery |
|------------|------------------------|-------------------|----------|
| User error | `odoo.exceptions.UserError` | `UserError` | Fix input, retry |
| Validation error | `odoo.exceptions.ValidationError` | `ValidationError` | Fix input, retry |
| Access error | `odoo.exceptions.AccessError` | `AccessError` | Check permissions |
| Missing record | `odoo.exceptions.MissingError` | `MissingError` | Verify record exists |
| Auth failure | `odoo.exceptions.AccessDenied` | `OdooAuthError` | Check API key, reconnect |
| Unhandled server error | Any other / generic traceback | `ServerError` | Inspect traceback |
| Network failure | fetch/socket error | `ConnectionError` | Check URL, retry |
| Input validation | Zod validation failure | `InputValidationError` | Fix tool arguments |

### Error Propagation

- **RPC Layer** (`rpc.ts`): Parses `JsonRpcResponse.error`, extracts `data.name` to determine exception class, throws appropriate `OdooError` subclass
- **Client Layer** (`client.ts`): Lets `OdooError` propagate. Catches network errors, wraps in `OdooConnectionError`
- **Tool Layer** (`tools/*.ts`): Catches `OdooError`, calls `formatMcpError()`, returns MCP `isError: true` response with structured JSON payload
- **MCP Layer**: MCP SDK handles transport serialization

### User-Facing Error Strategy

Errors returned to Claude as tool error content: `JSON.stringify(McpToolError)`. Odoo messages passed verbatim. Tracebacks included for `ServerError` only (useful for debugging but not user-facing — Claude decides what to show the end user).

### Tool Call Log Line


### Sanitization Rules

- `ODOO_API_KEY` value: NEVER logged anywhere
- `values` in `create`/`write`: fields matching `/password|credit_card|token|secret|api_key/i` replaced with `"[REDACTED]"`
- `domain` and `fields`: logged as-is (model names and field names are not PII)
- Log destination: stderr (stdout reserved for MCP stdio transport)

## Alternatives Considered

### Single Package

- **Description**: Ship `odoo-client` and MCP server as one npm package
- **Pros**: Simpler publish, one `package.json`
- **Cons**: Forces MCP SDK as dependency for anyone who just wants a JSON-RPC Odoo client. Larger install for non-MCP use cases.
- **Why rejected**: Two packages lets `odoo-client` serve as a standalone Odoo SDK. Clean dependency boundary.

### Module-Specific Semantic Tools

- **Description**: Tools like `accounting_post_entry`, `sales_create_order`
- **Pros**: Richer developer experience for common workflows
- **Cons**: Breaks on custom modules, non-standard localizations, future Odoo versions. Massive maintenance surface. Defeats the "works on any Odoo" value prop.
- **Why rejected**: Categorically rejected. Generic-only tool surface is the core design principle.

### HTTP/SSE Transport

- **Description**: Expose MCP server over HTTP for remote access
- **Pros**: Enables hosted deployment, multi-user
- **Cons**: Security surface (auth, rate limiting, TLS). Out of scope for v0. Stdio is what Claude Desktop/Code uses.
- **Why rejected**: v0 is stdio-only. HTTP/SSE deferred to future version.

### Session-Level Company State

- **Description**: Set active company once per session instead of per-call
- **Pros**: Less verbose tool calls
- **Cons**: Non-auditable. Hard to reason about state. Race conditions if Claude makes parallel calls.
- **Why rejected**: Stateless per-call is explicit, auditable, and race-free.
