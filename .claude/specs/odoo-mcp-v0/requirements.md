# Requirements: Odoo 19 MCP Server (v0)

> Generated by spec-engine v1.0.0

## Overview

An open-source Node.js MCP server exposing any Odoo 19 instance to Claude via 10 generic tools (ORM pass-through, universal invocation, introspection) and MCP resources (capability probe). Two npm packages under `@netlinks/`: `odoo-client` (JSON-RPC client, zero MCP dependency) and `odoo-mcp` (stdio MCP server, `npx`-executable). Generic-only tool surface — no module-specific semantic tools. Stateless multi-company context per call. MIT license.

## Depends On

None.

## User Stories

### US-1: Configuration Loading

**As an** OSS user (Odoo admin/developer)
**I want** the MCP server to load connection configuration from environment variables
**So that** I can configure it via `claude_desktop_config.json` without editing code

#### Acceptance Criteria (EARS Notation)

1. WHEN the server process starts THE SYSTEM SHALL read `ODOO_URL`, `ODOO_DB`, `ODOO_USERNAME`, and `ODOO_API_KEY` from environment variables
2. WHEN any of `ODOO_URL`, `ODOO_DB`, `ODOO_USERNAME`, or `ODOO_API_KEY` is missing or empty WHEN the server process starts THE SYSTEM SHALL exit with code 1 and print a structured error to stderr naming each missing variable
3. [inferred] WHEN `ODOO_URL` does not match the pattern `https?://.+` THE SYSTEM SHALL exit with code 1 and print a structured error to stderr indicating an invalid URL
4. [inferred] WHEN `ODOO_MCP_LOG_FILE` is set THE SYSTEM SHALL open the file path for append-mode writing and duplicate structured log lines to that file in addition to stdout/stderr
5. [inferred] WHEN `ODOO_MCP_LOG_FILE` is set but the path is not writable THE SYSTEM SHALL exit with code 1 and print a structured error to stderr indicating the log file is not writable
6. [threat-model] THE SYSTEM SHALL NOT include the value of `ODOO_API_KEY` in any stderr output, structured error message, or process exit message during configuration validation failure

### US-2: Odoo Authentication

**As an** OSS user
**I want** the server to authenticate against Odoo 19 using my API key at startup
**So that** all subsequent tool calls use a valid session

#### Acceptance Criteria (EARS Notation)

1. WHEN the server starts with valid configuration THE SYSTEM SHALL authenticate against the Odoo JSON-RPC endpoint using the configured credentials before registering any tools or resources
2. WHEN authentication succeeds THE SYSTEM SHALL store the session context (uid, session_id if cookie-based) for reuse across tool calls
3. WHEN authentication fails (invalid API key, unreachable server, wrong database) THE SYSTEM SHALL exit with code 1 and print a structured error to stderr containing the Odoo fault type and message
4. [inferred] WHEN a tool call receives an authentication error (e.g., revoked API key mid-session) THE SYSTEM SHALL return a structured MCP tool error with `error_type: "OdooAuthError"` and the Odoo-provided message
5. [threat-model] WHEN `ODOO_URL` uses the `http://` scheme (non-TLS) WHEN the server starts THE SYSTEM SHALL emit a structured warning to stderr indicating that credentials will be transmitted in plaintext

### US-3: Capability Probe and MCP Resources

**As** Claude (the agent)
**I want** the server to probe the connected Odoo instance at startup and expose the results as MCP resources
**So that** I have an instant mental model of the instance without issuing tool calls

#### Acceptance Criteria (EARS Notation)

1. WHEN authentication succeeds THE SYSTEM SHALL run the capability probe querying: installed modules (name, version), reports catalog (report_name, model, report_type), server-action catalog (name, model, type), companies (id, name, currency_id), currencies, fiscal year dates, user language, and user locale
2. WHEN the capability probe completes THE SYSTEM SHALL expose each probe result category as a separate MCP resource with a stable URI (e.g., `odoo://modules`, `odoo://reports`, `odoo://companies`)
3. WHEN a probe sub-query fails (e.g., `ir.actions.report` not accessible) THE SYSTEM SHALL populate that resource with an error payload `{ "error": "<message>" }` and continue probing remaining categories
4. THE SYSTEM SHALL NOT block tool registration if any individual probe sub-query fails
5. [inferred] WHEN a client reads an MCP resource THE SYSTEM SHALL return the cached probe result without re-querying Odoo
6. [threat-model] WHEN all capability probe sub-queries fail WHEN the server starts THE SYSTEM SHALL log a structured warning and populate every resource with an error payload; THE SYSTEM SHALL NOT serve empty arrays as if they were valid probe results

### US-4: ORM Tools (search_read, read, create, write, unlink, search_count)

**As** Claude (the agent)
**I want** six ORM pass-through tools that map directly to Odoo's ORM API
**So that** I can perform standard CRUD operations on any model in the connected instance

#### Acceptance Criteria (EARS Notation)

1. WHEN `odoo_search_read` is called with `model`, `domain`, `fields`, `limit`, `offset`, `order` THE SYSTEM SHALL execute `search_read` on the Odoo instance and return the result array
2. WHEN `odoo_read` is called with `model`, `ids`, `fields` THE SYSTEM SHALL execute `read` on the Odoo instance and return the result array
3. WHEN `odoo_create` is called with `model` and `values` (object or array of objects) THE SYSTEM SHALL execute `create` on the Odoo instance and return the created id(s)
4. WHEN `odoo_write` is called with `model`, `ids`, and `values` THE SYSTEM SHALL execute `write` on the Odoo instance and return the boolean result
5. WHEN `odoo_unlink` is called with `model` and `ids` THE SYSTEM SHALL execute `unlink` on the Odoo instance and return the boolean result
6. WHEN `odoo_search_count` is called with `model` and `domain` THE SYSTEM SHALL execute `search_count` on the Odoo instance and return the integer count
7. WHEN any ORM tool is called with a model that does not exist on the instance THE SYSTEM SHALL return a structured MCP tool error with `error_type` and `message` from the Odoo fault
8. WHEN any ORM tool is called with a malformed domain THE SYSTEM SHALL return a structured MCP tool error with the Odoo ValidationError details
9. [inferred] WHEN `odoo_search_read` is called without `limit` THE SYSTEM SHALL default to `limit: 80` to prevent unbounded result sets
10. [security] THE SYSTEM SHALL NOT pass tool input values to the JSON-RPC call without first validating them against the Zod input schema

### US-5: Universal Invocation Tools (execute, run_report, call_action)

**As** Claude (the agent)
**I want** three universal invocation tools for arbitrary method calls, report generation, and button actions
**So that** I can interact with any Odoo functionality beyond basic CRUD

#### Acceptance Criteria (EARS Notation)

1. WHEN `odoo_execute` is called with `model`, `method`, `args`, `kwargs` THE SYSTEM SHALL execute `execute_kw` on the Odoo instance with the provided arguments and return the raw result
2. WHEN `odoo_run_report` is called with `report_id` and `doc_ids` THE SYSTEM SHALL invoke the Odoo report engine and return the report output (base64-encoded binary content or structured data depending on report type)
3. WHEN `odoo_call_action` is called with `model`, `ids`, and `action_name` THE SYSTEM SHALL invoke the named action method on the specified records and return the action result
4. IF `context` is provided WHEN `odoo_call_action` is called THE SYSTEM SHALL merge the provided context with the multi-company context before invoking the action
5. WHEN `odoo_execute` is called with a method that raises `UserError` THE SYSTEM SHALL return a structured MCP tool error with `error_type: "UserError"` and the error message
6. WHEN `odoo_run_report` is called with a `report_id` that does not exist THE SYSTEM SHALL return a structured MCP tool error with the Odoo fault details
7. WHEN `odoo_call_action` is called with an `action_name` not defined on the model THE SYSTEM SHALL return a structured MCP tool error with the Odoo fault details
8. [inferred] WHEN `odoo_run_report` is called THE SYSTEM SHALL abstract over whichever report-engine protocol Odoo 19 exposes so the caller does not need to know the internal method name
9. [threat-model] THE SYSTEM SHALL validate that `model` parameters match the pattern `^[a-z][a-z0-9_.]*$` and `method` parameters match `^[a-z_][a-z0-9_]*$` before passing them to Odoo `execute_kw`

**As** Claude (the agent)
**I want** an introspection tool that returns field metadata for any model
**So that** I can discover the schema of models before querying them

#### Acceptance Criteria (EARS Notation)

2. [inferred] IF `attributes` is provided WHEN `odoo_fields_get` is called THE SYSTEM SHALL pass the attributes filter to limit returned metadata
3. WHEN `odoo_fields_get` is called with a model that does not exist THE SYSTEM SHALL return a structured MCP tool error with the Odoo fault details

### US-7: Multi-Company Context Threading

**As** Claude (the agent)
**I want** every tool to accept optional `allowed_company_ids` and `active_company_id` parameters
**So that** I can control multi-company read/write scope per call without server-side session state

#### Acceptance Criteria (EARS Notation)

1. THE SYSTEM SHALL accept `allowed_company_ids?: number[]` and `active_company_id?: number` as optional parameters on every tool
2. IF `allowed_company_ids` is not provided WHEN a tool is called THE SYSTEM SHALL default to the authenticated user's full `allowed_company_ids` from the session
3. IF `active_company_id` is not provided WHEN a tool is called THE SYSTEM SHALL default to the authenticated user's `company_id` from the session
4. WHEN a tool is called with multi-company context THE SYSTEM SHALL inject `allowed_company_ids` and `company_id` into the Odoo RPC `context` dict
5. THE SYSTEM SHALL NOT maintain any session-level company state between tool calls
6. WHEN a write operation is called with an `active_company_id` the user lacks access to THE SYSTEM SHALL return the Odoo `AccessError` as a structured MCP tool error
7. [threat-model] THE SYSTEM SHALL NOT allow caller-provided `context` to override the keys `uid`, `allowed_company_ids`, or `company_id` in the merged Odoo RPC context
8. [threat-model] WHEN a tool call includes `allowed_company_ids` containing a company ID not present in the session's `allowedCompanyIds` THE SYSTEM SHALL return an MCP tool error with `error_type: "InputValidationError"` before sending any RPC call to Odoo

### US-8: Error Propagation

**As** Claude (the agent)
**I want** Odoo RPC faults mapped to structured MCP tool errors
**So that** I can interpret errors and present them to the user without guessing

#### Acceptance Criteria (EARS Notation)

1. WHEN an Odoo RPC call returns a fault THE SYSTEM SHALL map it to an MCP tool error with structured payload `{ error_type: string, message: string, model?: string, method?: string, traceback?: string }`
2. THE SYSTEM SHALL map Odoo `UserError`, `ValidationError`, `AccessError`, `MissingError`, and unhandled server tracebacks to their respective `error_type` string values
3. THE SYSTEM SHALL NOT interpret or rewrite Odoo error messages — pass them through verbatim
4. [inferred] WHEN an Odoo RPC call fails due to a network error (timeout, connection refused, DNS failure) THE SYSTEM SHALL return an MCP tool error with `error_type: "ConnectionError"` and the underlying error message
5. [security] THE SYSTEM SHALL NOT expose internal error details (stack traces, file paths, database error messages) in user-facing output
6. [threat-model] THE SYSTEM SHALL NOT include Odoo tracebacks in MCP tool error payloads for any error type other than development-mode opt-in; the `traceback` field SHALL be omitted unless `ODOO_MCP_DEBUG=1` is set
7. [threat-model] THE SYSTEM SHALL set a request timeout of 30 seconds on all JSON-RPC HTTP calls to the Odoo instance and return an MCP tool error with `error_type: "ConnectionError"` if the timeout is exceeded

### US-9: Structured Logging

**As** the NETLINKS publisher
**I want** structured JSON log lines emitted per tool call
**So that** I can debug issues and monitor server behavior

#### Acceptance Criteria (EARS Notation)

1. WHEN a tool call completes (success or failure) THE SYSTEM SHALL emit a JSON log line to stderr containing `{ tool: string, args_sanitized: object, latency_ms: number, status: "ok" | "error", error?: string }`
2. THE SYSTEM SHALL NOT include the value of `ODOO_API_KEY` in any log output
3. [inferred] THE SYSTEM SHALL NOT include field values from `create`/`write` `values` payloads matching PII field names (`password`, `credit_card`, `token`, `secret`, `api_key`) in `args_sanitized`
4. WHILE `ODOO_MCP_LOG_FILE` is set THE SYSTEM SHALL write each log line to both stderr and the specified file
5. [inferred] WHEN the server starts THE SYSTEM SHALL emit a startup log line containing `{ event: "startup", odoo_url: string, odoo_db: string, odoo_username: string }`
6. [inferred] WHEN the server shuts down THE SYSTEM SHALL emit a shutdown log line containing `{ event: "shutdown" }`
7. [threat-model] WHEN `ODOO_MCP_LOG_FILE` is set THE SYSTEM SHALL create the log file with permissions `0o600` (owner read/write only) if it does not already exist

## Non-Functional Requirements

### Performance

1. THE SYSTEM SHALL complete the capability probe within 5 seconds for instances with up to 500 installed modules
2. THE SYSTEM SHALL add no more than 50ms of overhead per tool call beyond the Odoo RPC round-trip time
3. [inferred] THE SYSTEM SHALL NOT buffer tool call results in memory beyond the current request-response cycle

### Security

4. THE SYSTEM SHALL NOT log, print, or include `ODOO_API_KEY` in any output, error message, or MCP resource
5. THE SYSTEM SHALL NOT send telemetry, analytics, or any data to servers other than the configured `ODOO_URL`
6. [inferred] THE SYSTEM SHALL validate all tool input parameters against Zod schemas before making any Odoo RPC call

### Compatibility

7. THE SYSTEM SHALL run on Node.js 22 LTS with ESM module resolution
8. THE SYSTEM SHALL target Odoo 19.x only — earlier versions are explicitly unsupported
9. THE SYSTEM SHALL use TypeScript strict mode for all source files
10. THE SYSTEM SHALL use stdio as the only MCP transport

### Maintainability

11. [inferred] THE SYSTEM SHALL enforce consistent code style via Biome
12. [inferred] THE SYSTEM SHALL include Vitest unit tests for all `OdooClient` methods with mocked JSON-RPC responses
